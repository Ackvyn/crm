Cloudflare Email Routing → Worker

Inbound mail becomes tickets / notes when Cloudflare Email Routing delivers messages to your CRM Worker.

Outbound on free accounts (default)

The CRM Worker does not send mail unless you turn that on. Default keeps you on Cloudflare’s free Email Routing path (receive → Worker) without needing a paid sending product.

Agents reply from their normal client (mailto / Proton / Gmail / whatever). The console builds a message that CC or BCC’s your CRM inbound address (Settings → Sites → mailto copy mode). When that copy hits Email Routing → Worker, the thread is appended to the ticket — including agent-outbound notes when the sender’s From is on the site agent email allowlist.

Free-tier shape: receive on the Worker, send from the agent’s own mailbox, CC/BCC the CRM address so the ticket stays complete. See also Free tier & usage.

Optional: send from the Worker (paid Email Sending)

The Worker template includes a [[send_email]] binding (EMAIL). The Worker cannot detect whether your Cloudflare account has Email Sending enabled — that is a dashboard / plan setting on your account (Workers paid). In CRM → Settings → Sites, leave Send email from Worker off until you have:

  1. Workers paid plan with Email Sending enabled
  2. Sender domain verified for that mailbox
  3. [[send_email]] name = "EMAIL" deployed on your CRM Worker
  4. CRM inbound mailbox set (used as From)

Then flip the toggle on. Email customer opens an in-console composer and sends through the Worker; the ticket gets an agent-outbound note without needing CC. Mailto remains the path when the toggle is off.

Root domain vs subdomain

Ideal (Cloudflare’s happy path): Email Routing owns the root (apex) of the zone — MX/TXT as Cloudflare documents — and you create addresses that Send to a Worker. If you can put your day-to-day mail aliases on a subdomain instead (or move the alias app / password-manager “catch-all” off the apex), give Cloudflare the root and the dashboard will show Email Routing as fully enabled.

Common reality: the apex already points at another host (Proton, Google, Microsoft, …). Do not replace those root MX records with Cloudflare or you will break that mailbox.

In that case, use a subdomain only for CRM intake, e.g. crm.example.com → addresses like info@crm.example.com → Send to your CRM Worker.

Cloudflare’s UI may still say Email Routing is disabled or “not configured” for the zone when the root MX is not theirs. That warning is about apex ownership — it is not a hard block on subdomain routing. If the subdomain has the Email Routing MX/TXT records Cloudflare asks for, mail to those subdomain addresses still reaches a Worker. (Operators have run Proton on the apex and CRM on a subdomain this way successfully.)

Cleaner long-term: move personal/alias mail off the apex onto a subdomain (or another domain), then let Cloudflare Email Routing manage the root. Until you do that, subdomain → Worker is a valid, working setup — trust the DNS on the subdomain more than the scary “disabled” banner for the apex.

Setup checklist

  1. Cloudflare dashboard → Email → Email Routing for the zone.
  2. For subdomain intake: add the MX / TXT records Cloudflare shows for that hostname (keep apex MX on Proton/etc. unchanged).
  3. Create an address, e.g. info@crm.example.com, action Send to a Worker → select your CRM Worker.
  4. In Worker vars / settings, map that address to a site key (alias list or default inbound site).
  5. In CRM Settings → Sites, set the inbound mailbox to the same address so “Email customer” CC/BCC threads replies back into tickets.
  6. Add every agent’s From address to the agent email allowlist on that site. Inbound mail with an agent-outbound ticket marker is only credited as an agent note when From matches the list (customer mail is never blocked). Empty allowlist = never treat inbound as agent-outbound.

Optional vanity on apex

Keep customers writing info@example.com on your existing host, and forward that address → info@crm.example.com (Cloudflare → Worker). The short address stays on Proton (or similar); the CRM still sees the copy.
