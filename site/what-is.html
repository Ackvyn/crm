What is Ackvyn CRM?

An open, portable customer relationship stack for sites that already live in the free / static / git world.

Ackvyn CRM covers live chat, tickets, contacts, Documents (folders + PDF e-sign share links), macros, presence, inbound email, and Google reviews — without renting a separate SaaS CRM for every client site.

Same split you’ve seen in tools like Sveltia CMS: the app UI is served from a CDN; each operator deploys their own backend. Here the backend is a Cloudflare Worker (API, Durable Objects, WebSockets, Email Routing, git sync). Nobody shares another operator’s Worker.

Highlights

CDN-hosted GUI
Console, embed.js, and docs load from this site. Always the current UI; no bundling the admin SPA into every marketing site. Agents open the console from your operator site after sign-in — see Auth.
Your Worker for the API
Chat sockets, intake, presence, email, and encrypted crm-data/ commits run on your Cloudflare account.
Git-based records
Contacts, tickets, Documents library, macros, branding, and visitors live as encrypted blobs in your Pages/git repo. Reviews stay public JSON you can ship with the site.
Documents & e-sign
Encrypted file library with folders; place PDF fields and send share links. Signed copies attach to tickets; save into Documents to keep them after a ticket is deleted. See Documents & e-sign.
Drop-in embeds
One script tag on a client site: src → this CDN, data-crm → your Worker, data-site → the hub site key.
Contacts CSV
Import and export contacts with simple Email / Phone columns (compatible with Google Contacts and Outlook CSV).
Not a multi-tenant SaaS
You are the operator. Client sites talk to your Worker only. Data stays under your Cloudflare + GitHub (or equivalent) setup.
Built for static workflows
Aimed at freelancers and small shops already using Pages, git CMS, and free-tier hosting — the missing chat/CRM piece in that stack.

What it is not

  • Not a hosted chat product you log into on someone else’s domain and forget about infrastructure.
  • Not a replacement for a full enterprise helpdesk (Zendesk-class SLAs, phone, huge agent fleets).
  • Not “Ackvyn’s shared Worker for the internet.” Public docs never point traffic at a personal Worker URL.

Who it’s for

Developers and agencies who can deploy a Worker once, then hand clients a floating chat widget and keep tickets/contacts in their own git store. Editors and owners use the console from the operator site; they don’t need to touch Wrangler.
