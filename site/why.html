Story

Why

Static sites already had content figured out. Conversations were the missing open piece.

Free, statically hosted sites already have a workable path for content: GitHub (or similar), a static host, and a CDN-served git CMS — the pattern Sveltia CMS makes obvious. You keep the files; the editor is just a browser app.

Looking at what that stack made possible, the missing piece for many small-business sites wasn’t another page builder. It was live chat and a light CRM — contacts, tickets, presence, a way to answer email into the same record set — without bolting on a closed SaaS that owns the conversation data and doesn’t fit the “everything in git / free tier” workflow.

That gap felt reachable with the same headless idea: ship the GUI from a CDN, keep the API and data under the operator’s account, commit durable records into the site’s own repo. Ackvyn CRM is that experiment turned into a product shape.

The static-site workflow this assumes

  1. Marketing site on Pages (or similar), often free.
  2. Content edited via a git-based CMS (Sveltia-shaped or otherwise).
  3. Forms and light integrations that don’t need a traditional VPS.
  4. Still need someone to talk to visitors and track who said what — without abandoning the git/static model.

Steps 1–3 are solved in the open. Step 4 is where most people reach for a hosted chat widget.

About Tawk (and friends)

Tawk.to is, for a lot of small sites, the practical best alternative: free tier, solid widget, monitoring, agents, and years of product polish. In several dimensions it still has more capability than this project — and that’s fine to say out loud.

Ackvyn CRM is not trying to out-feature Tawk tomorrow. It’s the in-house, open, git-based alternative for operators who want chat + CRM records on the same footing as their CMS: UI from a CDN, backend they deploy, data in their store, no multi-tenant black box as the source of truth.

If you need the fullest free chat product with the least ops, use Tawk (or similar). If you want ownership and a Sveltia-shaped split — CDN GUI, your Worker, git for durable CRM state — that’s what this is for.

Built for a free Cloudflare account

The audience we care about most already has a Cloudflare zone for their domain and can run Workers on the Workers Free plan — including the familiar 100,000 requests per day allowance (per Worker; resets daily; confirm the current number in Cloudflare’s docs). The product shape assumes that budget is real, and that you may already be spending some of it on other Workers.

Sveltia CMS is the model for how far “client-side first” can go: the editor is a CDN script; the piece you usually deploy is a small auth Worker; after sign-in the browser talks to GitHub and commits content itself. Almost none of day-to-day CMS editing has to hit your Worker.

Ackvyn CRM aims at the same split where it can. The console loads from the CDN. Signed-in agents update and delete tickets, contacts, Documents, and related git-backed CRM state by committing encrypted files straight to your GitHub repo (Contents API) — not by round-tripping every edit through the CRM Worker. Auth stays on an auth Worker the same way a Sveltia-style CMS does.

Some paths cannot stay browser-only. Inbound email (Email Routing → Worker) and live chat (WebSockets, presence, Durable Objects) need a Worker API. Form intake that creates tickets without an agent session also lands on the Worker. Those are the expensive, always-on surfaces — so the design tries to keep everything else off that path and to avoid chatty API patterns that burn Free-plan requests for no reason.

The goal is a shop that can run fountain (or any small static site) with Cloudflare + this stack and stay inside Free-plan headroom for modest traffic. Mileage still varies: other Workers on the account, chat volume, email volume, and Cloudflare’s other Free limits (CPU, Durable Objects, and so on) all matter. See Free tier & Worker usage for the practical split.

What “success” looks like here

  • Operator deploys one Worker, sets up auth, and opens the console from their authenticated operator site.
  • Each client site gets a site key and an embed pointed at that Worker.
  • Chat and tickets land in the operator’s hub; reviews can publish to the site.
  • No public docs ever tell strangers to hit someone else’s Worker.
